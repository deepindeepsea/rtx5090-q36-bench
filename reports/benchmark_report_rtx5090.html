1 · What are we even measuring?

When you talk to an LLM, the work happens in two very different phases. Understanding them is the whole point of this report.

Prefill — "reading the prompt"

Before the model can answer, it has to read everything you gave it — your question, the documents, the chat history. It processes all of those input tokens at once, in parallel. This is called prefill (a.k.a. prompt processing, or pp).

Analogy: Prefill is a student skimming the entire exam paper before writing anything. They can read many lines at the same time. A fast reader gets to the first answer sooner — that's your "time to first token."

Prefill is compute-bound: it's limited by how many math operations (matrix multiplies) the GPU can crunch per second. A beefy GPU with thousands of cores shines here — if you keep those cores busy (more on that below).

Decode — "writing the answer"

Once the prompt is read, the model writes its reply one token at a time. Each new token depends on the one before it, so this part is sequential. This is decode (a.k.a. token generation, or tg).

Analogy: Decode is the student writing the answer word by word, and before writing each word they must re-skim their entire memory of the model. You can't write word #50 until word #49 exists.

Decode is memory-bandwidth-bound: to produce each single token, the GPU must read the model's active weights out of its memory. It's limited by how fast data moves, not by raw math. This is why decode is usually the slower, "streaming" part you watch tick by on screen.

Why both matter. Long prompt, short answer (summarizing a document, RAG, coding agents)? Prefill dominates your wait. Short prompt, long answer (chat, essays)? Decode dominates. A good engine has to win both.

Everything is measured in tokens per second (t/s) — higher is better. A token is roughly ¾ of a word.

2 · The setup

  • GPU: NVIDIA RTX 5090, 32 GB GDDR7 (~1.79 TB/s memory bandwidth), Blackwell architecture (compute capability 12.0, sm_120), driver 610.47.
  • Model: Qwen3.6-35B-A3B — 35.5B total parameters but a Mixture-of-Experts: only ~3B are actually used per token ("A3B" = active 3B). It's also a hybrid design — 10 classic attention layers + 30 lightweight recurrent (SSM / gated-DeltaNet) layers, which keeps its memory footprint small at long context.
  • Two weight formats tested: Q4_K_M (19.7 GB) and MXFP4 (20.6 GB) — both ~4-bit compressions of the model (see §5).
  • How we ran it: everything inside Docker containers on the GPU, so results are clean and reproducible. llama.cpp came from its official CUDA image; q36 we compiled from source in a CUDA 13.1 container.

Its own engine stat (from the server log) rose to ~5,177 t/s prompt-eval — proving the "2,500" was mostly measurement overhead (my client timer counts HTTP + tokenization that the engine doesn't). 4 · The single most important knob: micro-batch (-ub)

This one setting caused our biggest jump, so it's worth understanding.

When you feed the model a long prompt, the GPU doesn't process all the tokens in one giant step — it chops them into chunks. The micro-batch size is how many tokens go in each chunk.

Analogy — shoveling snow: The RTX 5090 has ~21,000 tiny "cores" (workers). The micro-batch is how big a scoop each worker takes. With the default scoop of 512, most of your 21,000 workers stand around idle. Bump the scoop to 8,192 and everyone's working at once — same snow, far fewer trips, nearly 2× the speed.
  • -ub (micro-batch): tokens crunched per GPU pass. This is the lever.
  • -b (batch): the bigger outer chunk fed to the scheduler; must be ≥ -ub.

Why doesn't this help decode? Because decode is one token at a time by nature — there's nothing to batch. Bigger scoops only help when there's a pile to move (prefill).

5 · The weight formats: Q4_K_M vs MXFP4

A raw 35B model is huge. To fit it in 32 GB and move it fast, the weights are quantized — compressed from 16-bit numbers down to ~4 bits each. We tested two 4-bit schemes:

  • Q4_K_M — llama.cpp's well-established 4-bit format. Great all-rounder.
  • MXFP4 — a newer 4-bit floating-point format that Blackwell GPUs can process with dedicated hardware. In theory, faster on a 5090.
The twist that explains q36. In llama.cpp, MXFP4 was no faster than Q4_K_M — the generic engine doesn't use Blackwell's native FP4 hardware paths. q36 does, with custom kernels. That's a big part of why q36 wins: it's not a better algorithm so much as hardware it actually knows how to drive.

7 · The ceiling: decode is capped by memory bandwidth

Here's the concept that predicts performance on any GPU before you run a thing. Because decode must stream the model's active weights from memory for every single token, its speed is:

max tokens/sec = memory bandwidth ÷ bytes read per token

For this model each token needs about 2.7 GB read from memory. That gives a hard "roofline" per GPU (and a realistic ~65% of it after real-world losses):

8 · What this means for the AMD endgame

The goal beyond today is porting q36's ideas to AMD ROCm on a Radeon AI Pro R9700 and a Strix Halo mini-PC. The roofline tells us what to expect before writing any code:

  • Radeon AI Pro R9700 (~640 GB/s): decode ceiling ≈ 234 t/s theoretical (~150 realistic) — roughly ⅓ of the 5090. Prefill (compute-bound) could still be strong.
  • Strix Halo (unified LPDDR5X, ~256 GB/s): decode ceiling ≈ 94 t/s (~60 realistic). Its superpower is capacity — it can hold very large models in unified memory — not raw speed.

Takeaway: porting the kernels won't make an AMD part decode like a 5090 — the memory bus sets the limit. The win on AMD will come from prefill and from fitting bigger models, not from out-decoding NVIDIA.

9 · The persistent lab (how to poke at it yourself)

q36 is built and waiting in a running container called q36dev. To jump in:

# open a shell inside the engine's container
docker exec -it q36dev bash
# re-run the throughput benchmark
./q36_bench -m /models/unsloth/Qwen3.6-35B-A3B-MXFP4_MOE.gguf -p 2048,8192 -n 128 -d 0,16384
# chat with the model interactively
./q36 -m /models/unsloth/Qwen3.6-35B-A3B-MXFP4_MOE.gguf
# or start its OpenAI-compatible server
./q36_server -m /models/unsloth/Qwen3.6-35B-A3B-MXFP4_MOE.gguf --port 8080 --ctx 32768

Tip: to watch any of our runs live, tail the log — Get-Content <logfile> -Wait -Tail 40 in PowerShell.
